# Supabase Egress Audit
**Date:** 2026-06-11  
**Scope:** Full codebase — all DB queries, polling patterns, storage usage, realtime subscriptions, and admin over-fetching  
**Database:** Supabase PostgreSQL (via `pg` pool + Drizzle ORM)  
**File Storage:** Cloudinary (100% — no Supabase Storage buckets in use)

---

## Executive Summary

Supabase egress in this platform is **entirely database-driven** — there is no Supabase Storage or Supabase Realtime in use. All bandwidth is generated by PostgreSQL queries executed against the Supabase-hosted instance via the connection pool.

Three categories account for the majority of egress:

1. **Notification polling** — every logged-in user hits `/api/notifications` (full list) every 60 s and `/api/notifications/unread-count` every 30 s. At scale this dominates.
2. **Admin over-fetching** — multiple admin panels independently load the full provider list (up to 500 rows × 2 KB each) with no server-side pagination. `ClientOperationsConsole` hardcodes `limit=500` users.
3. **Chat HTTP polling** — the messages page polls at 3 s / 5 s despite a working WebSocket already being open, generating redundant queries.

### Critical Duplicate: `/api/admin/monitoring/stats`

`MonitoringPanel` and `SystemBreaker` both use `queryKey: ["/api/admin/monitoring/stats"]` and `refetchInterval: 30_000`. React Query **deduplicates concurrent fetches** when both are mounted at the same time — but if they are mounted at different lifecycle points (e.g., SystemBreaker always visible, MonitoringPanel only when the monitoring tab is active), they can fire **independently**, generating 2× the database hits. This should be confirmed and consolidated.

## Admin Dashboard Over-Fetching

### `GET /api/admin/providers` — No Pagination

The most expensive admin pattern. **Five separate admin panels** each independently call this endpoint:

| Component | Purpose |
|-----------|---------|
| `ProviderOperationsConsole` | Full directory sidebar (displayed on mount) |
| `BookingsManagement` | Provider filter dropdown |
| `StaffOverview` | Staff listing |
| `CalendarView` | Provider picker |
| `AdminServiceRequests` | Service assignment |

The endpoint calls `storage.getAllProviders()` which executes `db.select()` (all columns) from `providers INNER JOIN users` with a hard cap of **500 rows**. Each row is ~2 KB (providers ~30 cols + users ~25 cols). At 100 providers: **200 KB/call**. At 500 providers: **1 MB/call**.

There is **no server-side caching** on the admin providers endpoint (only the public `/api/providers` is cached). Each component mount fires a fresh query.

### `GET /api/admin/users?limit=500&role=patient`

`ClientOperationsConsole` hardcodes `limit=500` in its fetch URL. The backend enforces a 500-row maximum on `getAllUsers()`. Every console mount downloads up to 500 full `users` rows (~250 KB).

### Queries with `staleTime: 0`

These bypass React Query's cache entirely and refetch on every mount:

| Endpoint | Component |
|----------|-----------|
| `/api/admin/service-requests` | `admin-service-requests.tsx` |
| `/api/appointments/:id` | `AppointmentActionDialog.tsx` |
| `/api/admin/sub-services` | `service-form-dialog.tsx` |
| `/api/provider/services` | `ProviderServicesTab.tsx` |

## WebSocket / Realtime Subscriptions

### WebSocket (Custom — no Supabase Realtime)

| Connection | URL | Events | Status |
|------------|-----|--------|--------|
| Chat messaging | `/ws/chat` | `message`, `typing`, `read`, `auto_reply` | Active |
| Slot availability | `/ws/slots` | Availability overrides during booking | Booking flow only |

**Key finding:** The chat WebSocket is fully functional and handles incoming message delivery. Despite this, `messages.tsx` **still polls** `/api/chat/messages` at 3 s and `/api/chat/conversations-rich` at 5 s in parallel. The WS invalidates React Query cache on receipt — the HTTP polls are redundant fallback behavior that was never removed.

### Supabase Realtime
**Not in use.** Zero `createClient` / Supabase Realtime channel subscriptions exist in the codebase. No Supabase Realtime bandwidth to optimize.

## Recommended Optimizations

### 🔴 HIGH Impact

---

#### H1 — Paginate `/api/admin/providers` (server-side)
**Saving: ~400 MB/month per 5 active admins** (scales with provider count)

Currently returns the full provider list on every mount. Add `limit`/`offset`/`search` to the admin endpoint and update all 5 consumer components to use server-side filtering.

```typescript
// server/routes/admin/admin-providers.routes.ts
app.get("/api/admin/providers", ..., async (req, res) => {
  const limit = Math.min(Number(req.query.limit) || 50, 100);
  const offset = Number(req.query.offset) || 0;
  const q = (req.query.q as string) || "";
  // Use searchProviders() instead of getAllProviders()
  const { rows, total } = await storage.searchProviders({
    q, countryCode: countryFilter, limit, offset
  });
  res.json({ providers: rows, total, page: Math.floor(offset/limit)+1, limit });
});
```

---

#### H2 — Remove HTTP chat polling when WebSocket is active
**Saving: ~1.6 GB/month per 100 chat-active users**

The WebSocket already invalidates React Query on message receipt. The HTTP polls in `messages.tsx` are pure redundancy.

```typescript
// client/src/pages/messages.tsx
// conversations-rich: remove refetchInterval, keep enabled
{ queryKey: ["/api/chat/conversations-rich"], staleTime: 30_000 }
// messages: remove refetchInterval — WS invalidates on delivery
{ queryKey: ["/api/chat/messages", selectedId], staleTime: 60_000 }
```

If WebSocket drops, React Query's `refetchOnWindowFocus: true` and a reconnect handler will recover. Add a `wsConnected` state to conditionally enable fallback polling only when WS is disconnected.

---

#### H3 — Fix duplicate `/api/admin/monitoring/stats` polling
**Saving: ~17 MB/month (the duplicate half), plus halved DB query load**

`SystemBreaker` and `MonitoringPanel` both independently poll the same endpoint every 30 s. Confirm that both components are never simultaneously mounted in separate React subtrees that don't share a Query Client. If they are, move `SystemBreaker` to read from the same `useQuery` call inside `MonitoringPanel` (pass data as a prop) rather than maintaining its own subscription.

---

#### H4 — Push unread count via WebSocket, remove polling
**Saving: ~96,000 DB queries/day per 100 DAU**

`/api/notifications/unread-count` is polled at 30 s from `header.tsx` and 60 s from `patient-dashboard.tsx` simultaneously. The chat WebSocket is already open for authenticated users. Emit a `notification:count` event over `/ws/chat` whenever a notification is dispatched, and update the React Query cache client-side:

```typescript
// server/services/channels/inapp.ts — after inserting user_notification:
broadcastToUser(userId, { type: "notification:count", count: newCount });

// client header.tsx — in WS message handler:
if (msg.type === "notification:count") {
  queryClient.setQueryData(["/api/notifications/unread-count"], { count: msg.count });
}
```

This eliminates the 30 s poll entirely for connected users.

**Saving: ~375 MB/month per 5 admin sessions**

```typescript
// client/src/components/admin/client-operations-console.tsx:912
fetch("/api/admin/users?limit=50&role=patient", ...)

```

#### M2 — Scope `/api/notifications` response to list columns only
**Saving: reduces payload ~40% per notification row**

The `data` JSONB column can be large (contains full appointment/booking details). For the bell-menu list view, only `id, type, title, message, isRead, createdAt` are displayed. Fetch full `data` only on the detail/action endpoint.

```typescript
// notification.routes.ts — GET /api/notifications
const rows = await db
  .select({
    id: userNotifications.id,
    type: userNotifications.type,
    title: userNotifications.title,
    message: userNotifications.message,
    isRead: userNotifications.isRead,
    createdAt: userNotifications.createdAt,
  })
  .from(userNotifications)
  .where(eq(userNotifications.userId, req.user!.id))
  .orderBy(desc(userNotifications.createdAt))
  .limit(limit);
```

#### M4 — Add explicit column lists to high-volume `SELECT *`

Priority order (highest row count risk):

```sql
-- availability_exceptions: only date, start_time, end_time, reason, is_recurring
SELECT id, provider_id, date, start_time, end_time, reason, is_recurring
  FROM availability_exceptions WHERE provider_id = $1 ORDER BY date ASC

-- patient_gallery: only id, user_id, image_url, caption, created_at
SELECT id, user_id, image_url, caption, created_at
  FROM patient_gallery WHERE user_id = $1 ORDER BY created_at DESC

-- patient_notes: omit large free-text body from list view
SELECT id, provider_id, patient_id, title, created_at, updated_at
  FROM patient_notes WHERE provider_id = $1 AND patient_id = $2 ORDER BY created_at DESC
```

#### M5 — Reduce `/api/admin/monitoring/events` LIMIT from 500 to 100
**Saves ~250 KB per poll × 2,880 polls/day = ~720 MB/month**

The monitoring panel table only renders the most recent ~20 rows visibly. Loading 500 rows is 5× more than the UI can display.

```typescript
// admin-monitoring.routes.ts
LIMIT 100  // was 500
```

---

#### M6 — Remove patient-dashboard duplicate unread poll
`patient-dashboard.tsx` polls `/api/notifications/unread-count` every 60 s but `header.tsx` already polls the same endpoint every 30 s. The header result is in the shared React Query cache — the patient dashboard can read from `useQuery` without a separate `refetchInterval`.

```typescript
// patient-dashboard.tsx
const { data: unreadData } = useQuery({
  queryKey: ["/api/notifications/unread-count"],
  // Remove refetchInterval: 60_000 — share header.tsx's subscription
});
```
