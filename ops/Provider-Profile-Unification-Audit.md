## 2. Duplications & Dead Code Found

### Document Management Overlap
Three UI surfaces target the same backend endpoints (`/api/provider/documents`):
- `ProviderKYC.tsx` — within provider dashboard
- `ProviderDocumentsPanel.tsx` — full credential manager
- `provider-setup.tsx` step 3 — onboarding upload

**Status:** Intentional three-tier design (onboarding → ongoing management → admin review). No dead code removed — each serves a distinct user moment. Documented as expected duplication.

### Booking Window Settings Overlap
- `profile.tsx` Provider tab — minimum notice + booking window
- `SmartScheduler.tsx` Settings tab — same fields

**Status:** Both write to the same `providers` columns. Pre-launch; left for P9 UX unification pass.

### `/api/providers/me` Alias
Explicit alias of `/api/provider/me` for backward compatibility. Clean explicit alias; no hidden divergence found.

### Admin Document Verification
`ProviderReviewQueue` and `DocumentQueue` both show approve/reject actions. This is intentional: `ProviderReviewQueue` is the onboarding gate (profile + docs together); `DocumentQueue` is the ongoing operational queue. `ProviderOperationsConsole` is explicitly view-only for documents, delegating decisions to `DocumentQueue` per the existing design.

## 4. Delivery Mode Audit

### Architecture

**Two-level model (correct, no changes needed to the model itself):**

1. **Provider level** — `providers.serviceModes: text[]`  
   Global capability flags: `["online", "home_visit", "clinic_visit"]`

2. **Service level** — `services.locationMode: text`  
   Per-service restriction: `clinic_only | home_only | online_only | both | all | clinic_online | home_online`

**Booking flow enforcement** (`book-wizard.tsx`):  
Auto-corrects `visitType` if selected mode conflicts with `service.locationMode`. Backend enforces matching fees at appointment creation. ✅ Correct.

### Bug 1: Provider Card — Legacy `homeVisitFee` Fallback (FIXED)

**Before:**
```typescript
const toShow = modes.length > 0
  ? modeConfigs.filter((m) => modes.includes(m.key))
  : modeConfigs.filter((m) => {
      if (m.key === "online") return true;
      if (m.key === "home_visit") return !!(provider as any).homeVisitFee;
      return false;
    });
```

When `serviceModes` was empty, the card showed "Online" unconditionally and "Home Visit" if `homeVisitFee` was set — bypassing the canonical `serviceModes` field entirely.

**After:**
```typescript
const toShow = modeConfigs.filter((m) => modes.includes(m.key));
```

Mode badges now show only what `serviceModes` declares. If `serviceModes` is empty, no mode badges are shown.

### Bug 2: Provider Listing Filter — `homeVisitFee` Instead of `serviceModes` (FIXED)

**Before (`client/src/pages/providers.tsx`):**
```typescript
if (filters.homeVisit && !p.homeVisitFee) return false;
```

The "Home Visit Available" filter checked for a non-null `homeVisitFee` column — a provider-level legacy fee field — instead of the canonical `serviceModes` array.

**After:**
```typescript
if (filters.homeVisit && !(p as any).serviceModes?.includes("home_visit")) return false;
```

Filter now correctly uses `serviceModes`.

### Remaining Legacy Fields (Documented, Not Removed)

`providers` table still carries `consultationFee`, `homeVisitFee`, `telemedicineFee` as top-level columns alongside the per-service fee columns on `services`. The booking engine already prefers per-service fees; the provider-level columns are fallbacks used by the provider matcher and analytics. Removing them requires a Supabase migration and analytics recalibration — deferred to P9 with a tracked recommendation.

## 8. Remaining Recommendations (P9)

1. **Merge duplicate booking-window settings** — `profile.tsx` and `SmartScheduler.tsx` both expose `minimumNoticeMinutes` / `maximumBookingDays`. Consolidate to one location (SmartScheduler) and remove from profile.tsx provider tab.

2. **Deprecate provider-level fee columns** — `providers.consultationFee`, `providers.homeVisitFee`, `providers.telemedicineFee` are now redundant with per-service fees. Plan a Supabase migration to remove them and update the provider matcher to read from service rows only.

3. **Enforce mobile number at submit-review** — Currently a UI soft-check only. Wire `mobileVerified: true` into the `submit-review` schema gate once Twilio is active.

4. **Consolidate `ProviderKYC.tsx` and `provider-documents-panel.tsx`** — These overlap significantly. One unified document manager component would simplify the codebase.

5. **Naming consistency** — `serviceType` is used in search components to mean provider *profession/category* (doctor, physio, nurse) but `visitType` means delivery mode. Rename the search filter to `providerType` to eliminate the confusion.

## Final Architecture (Target State — Achieved for P8)

```
ONE providers table        — identity, credentials, serviceModes, addresses, status
ONE provider_documents     — KYC docs (id_card REQUIRED, address_proof REQUIRED, insurance OPTIONAL)
ONE provider_credentials   — public-facing verified certs
ONE services table         — per-service config including locationMode + per-mode fees
ONE POST /api/provider/setup  — profile write endpoint
ONE POST /api/provider/submit-review — KYC gate with correct required docs
ONE serviceModes[]         — canonical source for delivery mode display and filtering
```
